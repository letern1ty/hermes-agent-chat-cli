## 🧠 三种高级模式对比

| 模式 | 核心思想 | 适用场景 | 优点 | 缺点 |
|------|----------|----------|------|------|
| **ReAct** | Thought-Action-Observation 循环 | 需要多步推理的问题 | 减少幻觉，可追溯 | 可能较慢 |
| **规划** | 先拆解任务，再执行 | 复杂多步骤任务 | 清晰、可监控 | 规划可能不完整 |
| **多 Agent** | 多个专家 Agent 协作 | 需要多领域知识 | 专业、可并行 | 系统复杂、成本高 |

## 🚀 Day 6: ReAct 模式

### 核心思想

```
传统方式：
用户问 → LLM 直接回答（可能瞎编）

ReAct 方式：
用户问 → 思考 → 行动 → 观察 → 思考 → 行动 → ... → 回答
```

### 示例问题

```
"周杰伦出生到现在多少年了？"

ReAct 流程：
1. Thought: 需要知道周杰伦的出生年份和当前年份
2. Action: 调用 search_tool("周杰伦 出生年份")
3. Observation: 1979 年 1 月 18 日
4. Action: 调用 get_current_year()
5. Observation: 2026 年
6. Thought: 现在可以计算了
7. Answer: 周杰伦现在 47 岁
```

## 📋 Day 7: 规划模式

### 核心思想

```
阶段 1 - 规划：
用户任务 → LLM 拆解成步骤列表

阶段 2 - 执行：
对每个步骤 → 调用工具执行 → 收集结果
```

### 示例任务

```
"帮我初始化一个 React 项目"

规划：
1. 创建项目目录 my-react-app
2. 运行 npm init -y
3. 安装 React 依赖
4. 创建 index.html
5. 创建 src/index.js
6. 创建 src/App.jsx

执行：
依次执行每个步骤...
```

## 🤖 Day 8: 多 Agent 协作

### 架构

```
用户任务
   │
   ▼
协调者（Coordinator）
   │
   ├──→ 代码审查 Agent（检查代码质量）
   ├──→ 安全专家 Agent（检查漏洞）
   ├──→ 测试工程师 Agent（制定测试计划）
   │
   ▼
整合结果 → 最终报告
```

### 示例场景

```
代码审查任务：

CodeReviewer:
- 检查代码风格
- 检查性能问题
- 提出改进建议

SecurityExpert:
- 检查 SQL 注入
- 检查 XSS 漏洞
- 检查认证问题

TestEngineer:
- 制定单元测试计划
- 制定集成测试计划
- 生成测试用例

Coordinator:
- 整合所有报告
- 生成最终审查结果
```

## 💡 如何选择模式？

| 你的需求 | 推荐模式 |
|----------|----------|
| 简单问答 | 基础工具调用 |
| 需要推理（如数学题） | ReAct |
| 多步骤任务（如部署项目） | 规划模式 |
| 需要多领域专业知识 | 多 Agent |
| 追求速度 | 基础工具调用 |
| 追求准确性 | ReAct 或规划模式 |
