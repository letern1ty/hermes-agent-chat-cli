## 概念 1：ReAct 模式（推理 + 行动）

### 核心思想

```
传统方式：
用户问 → LLM 直接回答（可能瞎编）

ReAct 方式：
用户问 → 思考 → 行动 → 观察 → 思考 → 行动 → ... → 回答
```

### 关键洞察

1. **思考（Thought）**：LLM 分析当前情况，决定下一步
2. **行动（Action）**：调用工具获取信息
3. **观察（Observation）**：查看工具返回结果
4. **循环**：直到有足够信息

### 为什么有效？

- 强制 LLM "慢思考"，减少幻觉
- 每一步都有依据，可追溯
- 适合需要多步推理的问题

### 运行示例

```bash
cd ~/my-first-agent
source venv/bin/activate
python step6_react.py
```

### 典型问题

```
- "周杰伦出生到现在多少年了？"
  → 需要：搜索出生年份 → 获取当前年份 → 计算

- "特斯拉 CEO 是哪里人？"
  → 需要：搜索 CEO 是谁 → 搜索他的出生地
```

## 概念 2：规划模式（Plan-and-Execute）

### 核心思想

```
两阶段流程：

阶段 1 - 规划：
用户任务 → LLM 拆解成步骤列表

阶段 2 - 执行：
对每个步骤 → 调用工具执行 → 收集结果
```

### 为什么需要规划？

| 无规划 | 有规划 |
|--------|--------|
| LLM 可能迷失方向 | 清晰的执行路径 |
| 容易遗漏步骤 | 完整的任务覆盖 |
| 难以追踪进度 | 每步可监控 |

### 运行示例

```bash
python step6_planning.py
```

### 典型任务

```
- "帮我初始化一个 React 项目"
  计划：[创建目录 → 初始化 npm → 安装依赖 → 创建文件]

- "检查项目状态并修复问题"
  计划：[列出文件 → 运行测试 → 运行 lint → 修复问题]
```

## 概念 3：多 Agent 协作

### 核心思想

```
用户任务
   │
   ▼
协调者（Coordinator）
   │
   ├──→ 代码审查 Agent
   ├──→ 安全专家 Agent
   ├──→ 测试工程师 Agent
   │
   ▼
整合结果 → 最终报告
```

### 为什么多 Agent？

| 单 Agent | 多 Agent |
|----------|----------|
| 通用但不够深入 | 每个领域专家级 |
| 上下文容易混乱 | 职责清晰分离 |
| 难以并行 | 可以并行执行 |

### 运行示例

```bash
python step6_multi_agent.py
```

### 典型场景

```
代码审查任务：
- CodeReviewer：检查代码质量、性能
- SecurityExpert：检查安全漏洞
- TestEngineer：制定测试计划
- Coordinator：整合所有结果
```

## 三种模式对比

| 模式 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **ReAct** | 需要多步推理的问题 | 减少幻觉，可追溯 | 可能较慢 |
| **规划** | 复杂多步骤任务 | 清晰、可监控 | 规划可能不完整 |
| **多 Agent** | 需要多领域专业知识 | 专业、可并行 | 系统复杂 |

## 常见问题

### Q: ReAct 和普通工具调用有什么区别？

A: ReAct 强调"思考"过程，LLM 需要解释为什么调用这个工具。普通工具调用只关注结果。

### Q: 规划模式会不会很慢？

A: 是的，多了一轮规划。但对于复杂任务，总体效率更高（少走弯路）。

### Q: 多 Agent 需要多个 API 调用吗？

A: 是的，每个 Agent 独立调用 LLM。成本更高，但质量也更好。

### Q: 我应该用哪种模式？

A: 
- 简单问答 → 基础工具调用
- 需要推理 → ReAct
- 复杂任务 → 规划模式
- 多领域 → 多 Agent
